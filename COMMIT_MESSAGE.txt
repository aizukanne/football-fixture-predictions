Fix: Critical bugs in API calls, tactical analysis, and caching

This commit addresses multiple critical issues discovered during team parameter calculation:

## 1. Season Parameter Required for Fixtures API (CRITICAL)
- Added required `season` parameter to get_fixtures_goals()
- Updated api_client.py (lines 616, 637, 957) to accept and pass season
- Updated tactical_analyzer.py (lines 646, 828) to provide season parameter
- **Impact**: Fixes 0 fixtures being returned for ALL leagues
- **Test**: All leagues now return fixtures (Premier League: 380, Championship: 557, Eliteserien: 113)

## 2. get_league_teams() Type Mismatch Fixes
- Fixed tactical_analyzer.py line 936: Removed incorrect 'response' key check on list
- Fixed tactical_analyzer.py line 940: Changed `teams_data['response']` to `teams_data`
- Fixed tactical_analyzer.py line 950: Changed `team_data.get('team', {}).get('id')` to `team_data.get('team_id')`
- **Impact**: Resolves "No league teams data" warnings and enables formation analysis
- **Root Cause**: get_league_teams() returns list, not dict with 'response' key

## 3. fixture_events_cache Table Creation
- Added create_fixture_events_cache_table() method to deploy_tables.py (lines 429-492)
- Added table to deployment list (line 521)
- Proper table schema: fixture_id (String) primary key, 7-day TTL
- **Impact**: Enables event caching to reduce API calls

## 4. Hardcoded Table Name Fix
- Fixed database_client.py line 990: Use _get_table_name() instead of hardcoded string
- **Impact**: Table name now correctly applies prefix/suffix (football_fixture_events_cache_prod)
- **Before**: 'fixture_events_cache' (AccessDenied errors)
- **After**: 'football_fixture_events_cache_prod' (matches IAM permissions)

## 5. DynamoDB Query Type Conversion
- Fixed database_client.py line 629: Convert season to int() for DynamoDB NUMBER type
- **Impact**: Classification queries now return matches instead of 0 results
- **Root Cause**: DynamoDB stores season as NUMBER but was queried as STRING
